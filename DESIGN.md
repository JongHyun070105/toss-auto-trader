# Batter’s Eye 디자인 시스템 재정비안

## 1) 경험 원칙

1. **다크 퍼스트, 라이트 호환**  
   브랜드 감성은 다크 테마를 기준으로 설계하고, 라이트 모드는 같은 구조를 밝게 번역한 보조 모드로 운영한다.

2. **한 화면 = 한 목적**  
   각 화면은 “이해 → 행동”이 즉시 이어져야 한다. 주요 CTA는 1개만 둔다.

3. **훈련 우선, 장식 최소화**  
   홈도 결국 다음 훈련으로 이어져야 한다. 분석보다 행동 전환이 먼저다.

4. **메트릭은 항상 액션과 함께**  
   정확도·반응속도·약점은 숫자로 끝내지 말고 “다음에 무엇을 할지”까지 붙인다.

5. **코치 톤 앤 매너 유지**  
   문장은 짧고 직접적이어야 한다. 설명문보다 코치 멘트처럼 말한다.

6. **게임 감성은 훈련 순간에만 강하게**  
   투구 시작, 정답 공개, 결과 요약에는 긴장감 있는 모션을 주되 전체 앱은 차분하게 유지한다.

7. **진행 상태는 항상 보이게**  
   온보딩, 레벨 테스트, 세션, 결과 저장 등 모든 플로우에는 현재 단계와 남은 양을 표시한다.

8. **AI 개인화는 설명 가능해야 한다**  
   추천은 반드시 근거(레벨 테스트 결과, 최근 약점, 목표)와 함께 노출한다.

9. **반복 사용을 고려한 일관성**  
   모든 화면은 동일한 헤더 구조, 카드 규칙, CTA 계층을 사용한다. 일회성 위젯을 줄인다.

10. **모바일 스포츠 앱답게 터치 친화적이어야 한다**  
    주요 터치 영역은 충분히 크게, 선택지는 짧고 빠르게 누를 수 있게 만든다.

11. **디자인은 한 시스템처럼 보여야 한다**  
    `awesome-design-md` 레퍼런스처럼 온보딩, 인증, 홈, 훈련, 결과, 설정이 모두 같은 제품군처럼 보여야 한다.

12. **AI와 3D는 기능이 아니라 경험의 핵심이다**  
    개인화는 Gemini 2.5 Flash Lite 기준으로 설계하고, pitch scene의 현실감은 별도 렌더러까지 고려해 구조를 잡는다.

## 2) 비주얼 시스템

### 2-1. 컬러 역할

- **Background**: `#06111F`, `#07111F`, `#0B1730`
- **Surface**: `#10233C`, `#162A4A`
- **Primary Accent**: `#62E6FF`
- **Success / Correct**: `#84F58E`
- **Insight / Warning**: `#FFCF72`
- **Error / Miss**: `#FF6B7A`
- **Text Primary**: `#F5F8FF`
- **Text Secondary**: `rgba(255,255,255,0.72)`
- **Text Muted**: `rgba(255,255,255,0.52)`
- **Border**: `rgba(255,255,255,0.08)`

#### 컬러 사용 규칙
- 화면당 강한 포인트 컬러는 1개만 쓴다.
- 상태 표현은 **색 + 아이콘/텍스트**를 함께 사용한다.
- 훈련 모드별 보조 포인트는 허용하되 기본 구조색은 유지한다.
- 결과 화면만 성공/실패 대비를 조금 더 강하게 쓴다.

### 2-2. 타이포그래피

- **Display / H1**: 32 / 800~900 / -0.03em
- **H2**: 24 / 700~800 / -0.02em
- **Section Title**: 18 / 700
- **Body**: 16 / 400 / line-height 1.45~1.5
- **Label / Caption**: 12~13 / 600

#### 타이포 규칙
- 한국어 카피를 기본으로 두고, 영어는 모드명·보조 라벨에 제한한다.
- 제목은 1~2줄, 본문은 3줄 이내를 기본값으로 잡는다.
- 큰 제목 다음에는 항상 짧은 코치 문장 또는 설명 1개만 붙인다.

### 2-3. 간격, 라운드, 깊이

- **8pt 그리드** 사용
- 화면 좌우 패딩: **20px**
- 섹션 간 간격: **16~24px**
- 카드 내부 패딩: **18~20px**
- Hero Card radius: **28px**
- Default Card radius: **24px**
- Button / Input radius: **18px**
- Chip radius: **999px**
- 그림자보다 **얇은 보더 + 톤 차이**로 레이어를 만든다.

### 2-4. 모션

- 탭/선택 반응: **120~160ms**
- 카드 전환/스위치: **200~240ms**
- 화면 전환: **260~320ms**
- 투구 리빌: **600~700ms**
- 결과 피드백은 짧고 명확하게, 과한 파티클/네온 효과는 금지

## 3) 공통 레이아웃 규칙

### 3-1. 화면 기본 구조
모든 주요 화면은 아래 순서를 기본으로 한다.

1. **Page Header**: 타이틀 + 한 줄 설명/코치 멘트
2. **Hero / Primary Block**: 오늘의 추천, 현재 단계, 결과 요약 등
3. **Secondary Blocks**: 메트릭, 카드 목록, 선택지
4. **Primary CTA**: 다음 행동 1개
5. **Supportive Info**: 도움말, 설정, 세부 정보

### 3-2. CTA 계층
- **Primary CTA**: 화면당 1개, 전체 폭 또는 강한 시각 우선순위
- **Secondary CTA**: 보조 행동, tonal/outlined 형태
- **Tertiary CTA**: 텍스트 버튼, 뒤로가기/자세히 보기 정도에만 사용

### 3-3. 상태 규칙
모든 화면은 아래 상태를 고려한다.
- Loading
- Empty
- Error
- Completed
- Returning user

## 4) 핵심 컴포넌트 정의

### 4-1. Hero Card
용도: 추천 훈련, 레벨 결과, 세션 완료 요약  
구성: 상태 Pill / 제목 / 한 줄 코치 멘트 / 설명 / Primary CTA

### 4-2. Mode Card
용도: 훈련 모드 선택  
구성: 아이콘 / 모드명 / 짧은 설명 / 예상 효과 / CTA 또는 진입 affordance

### 4-3. Metric Card
용도: 정확도, 반응속도, 세션 수, 약점 존  
규칙: 숫자만 보여주지 말고 해석 가능한 라벨을 붙인다.

### 4-4. Coach Card
용도: AI 추천, 다음 훈련 팁, 오류 복구 안내  
규칙: 2~3문장 이내, 행동 중심 문장으로 작성한다.

### 4-5. Progress Strip
용도: 온보딩, 레벨 테스트, 훈련 라운드  
규칙: 현재 단계와 전체 단계가 함께 보여야 한다.

### 4-6. Choice Component
- 2~4개 선택: segmented / chips 우선
- 5개 이상: list card 또는 bottom sheet
- 입력보다 선택을 우선해서 속도를 높인다.

### 4-7. Result Insight Block
용도: 결과 해석  
구성: 핵심 수치 / 약점 / 원인 / 다음 훈련 추천

## 5) 화면 정보 구조

## 5-1. 최상위 구조

### A. 온보딩 스택
`온보딩 인트로 → 인증 → 기본 프로필 → 레벨 테스트 → AI 개인화 → 메인 앱`

### B. 메인 앱 쉘
하단 탭 4개를 기준으로 고정한다.
- **홈**
- **훈련**
- **결과**
- **설정**

> 온보딩은 전체 화면 플로우, 메인 앱은 탭 기반 구조로 분리한다.  
> 현재처럼 홈에 기능을 과도하게 몰아넣는 방식은 지양한다.

## 5-2. 온보딩

### 목적
사용자 의도 파악, 계정 생성, 실력 측정, 첫 개인화 플랜 제시

### 권장 단계
1. **온보딩 인트로**
   - 앱 가치 제안
   - 3분 세션 설명
   - “지금 시작” CTA
2. **인증**
   - 가입 / 로그인
   - 왜 계정이 필요한지 짧게 설명
3. **기본 프로필**
   - 이름, 나이, 포지션, 타석 방향, 경험, 목표
   - 입력 수를 최소화하고 선택형 UI 우선
4. **레벨 테스트**
   - 진행률 표시
   - 짧은 라운드 단위 피드백
   - 마지막에 레벨 산정
5. **AI 개인화**
   - 당신의 현재 레벨
   - 핵심 강점 1개
   - 핵심 약점 1개
   - 첫 7일 추천 플랜
   - “이 플랜으로 시작” CTA
   - 기본 모델 타깃: Gemini 2.5 Flash Lite
   - API 키 부재 시 동일한 UI 구조의 fallback plan 사용

### 온보딩 완료 조건
- 계정 생성 완료
- 프로필 저장 완료
- 레벨 결과 저장 완료
- 초기 AI 플랜 생성 완료

## 5-3. 홈

### 역할
오늘 무엇을 해야 하는지 가장 빠르게 보여주는 허브

### 권장 블록 순서
1. **오늘의 포커스 Hero**
   - 추천 모드
   - 코치 멘트
   - 바로 시작 CTA
2. **오늘의 지표 요약**
   - 최근 정확도
   - 평균 반응속도
   - 누적 세션 / 연속 일수
3. **AI 추천 카드**
   - 왜 이 훈련을 추천하는지
4. **최근 결과 스냅샷**
   - 가장 최근 세션 1개만 요약
5. **주간 계획 / 다음 단계**
   - 7일 플랜 또는 다음 목표

### 홈에서 하지 말 것
- 설정 정보 과다 노출
- 긴 설명문
- 모든 훈련 모드를 동등한 비중으로 한 화면에 과다 배치

## 5-4. 훈련

### 역할
모드 선택부터 실제 세션 플레이까지 담당

### 구조
- **훈련 홈**
  - 모드 목록
  - 추천 모드 강조
  - 모드별 훈련 목적 설명
- **세션 준비 화면**
  - 난도 / 라운드 수 / 목표 안내
- **진행 중 화면**
  - 현재 라운드
  - Pitch Scene
  - 입력 선택지
  - 즉시 피드백
- **세션 종료 후 결과로 이동**

### 훈련 화면 규칙
- 정보 밀도보다 반응 속도를 우선한다.
- 시선은 항상 “공 → 선택지 → 피드백 → 다음” 순서를 따른다.
- 게임 감성은 이 구간에서만 가장 강하게 표현한다.
- Pitch Scene은 점수판이 아니라 **인지 훈련 장면**처럼 보여야 한다.
- 3D 현실감은 아래 우선순위로 개선한다.
  1. 깊이감 있는 카메라/원근/블러/그림자
  2. 릴리스 포인트와 포수 시점의 거리감 강화
  3. 공속/회전/궤적 차이를 시각적으로 분리
  4. 필요 시 pitch renderer만 Unity 모듈로 분리 검토
- 단, 어느 단계에서도 선택지 입력 속도와 반응성은 희생하지 않는다.

## 5-5. 결과

### 역할
훈련이 끝난 뒤 “무엇이 좋았고, 무엇을 다음에 보완할지”를 명확히 보여준다.

### 구조
- **이번 세션 요약 Hero**
- **핵심 메트릭 3~4개**
- **약점 분석**
- **AI 코치 해석**
- **다음 추천 훈련 CTA**
- **히스토리 / 추세 보기**

### 결과 탭과 세션 결과 화면의 관계
- 세션 직후 결과 화면은 **즉시 피드백용**
- 메인 탭의 결과 화면은 **누적 기록/비교용**

## 5-6. 설정

### 역할
계정, 표시 옵션, 알림, 데이터 관련 제어

### 구조
1. 계정 / 프로필
2. 테마 / 언어
3. 알림 / 훈련 리마인더
4. 데이터 / 개인정보 / 로그아웃

### 규칙
- 설정은 홈에서 분리한다.
- 설명은 짧게, 선택지는 명확하게 노출한다.
- 미리보기 카드가 있으면 현재 상태를 한 줄로 요약한다.

## 6) 핵심 사용자 플로우

### 첫 사용자 플로우
`인트로 → 인증 → 프로필 → 레벨 테스트 → AI 개인화 → 홈 → 추천 훈련 시작 → 결과 확인`

### 재방문 사용자 플로우
`앱 실행 → 홈 Hero → 바로 훈련 → 결과 → AI 추천 갱신`

### 실패 복구 플로우
- 인증 실패: 원인 + 재시도 CTA
- 저장 실패: 재시도 / 임시 저장 상태 표시
- 네트워크/AI 실패: 기본 추천 플랜 fallback 제공

## 7) 카피 원칙

- 코치처럼 짧고 직접적으로 쓴다.
- 숫자는 해석과 같이 쓴다. 예: `반응속도 420ms · 높게 뜨는 공 판단이 늦었어.`
- 긍정 피드백도 과장하지 않는다.
- 지나치게 일반적인 생산성 앱 문구는 피한다.
- 한국어 우선, 영어는 브랜드 감도 보조 정도로 제한한다.

## 8) Flutter 적용 가이드

- 공통 토큰은 `app_theme.dart` 중심으로 통합한다.
- radius, spacing, gradient, text color를 화면별 상수로 흩뿌리지 않는다.
- 공통 카드/헤더/메트릭 위젯을 만들어 재사용한다.
- 온보딩 스테이지와 메인 탭 구조를 라우팅 레벨에서 분리한다.
- 홈/훈련/결과/설정은 같은 앱 쉘 규칙을 따른다.

## 9) 반드시 피할 것

- 화면마다 다른 헤더 스타일
- 임의의 그라디언트 추가
- 한 화면에 Primary CTA 여러 개 배치
- 결과 없는 AI 추천
- 색만으로 성공/실패를 표현하는 UI
- 홈에 설정/프로필/기록/훈련을 모두 과밀하게 적재하는 구조

## 10) 문서 유지 규칙

디자인 변경 시 아래를 함께 업데이트한다.
- 정보 구조가 바뀌면: 이 문서의 `화면 정보 구조`
- 컴포넌트가 추가되면: `핵심 컴포넌트 정의`
- 새로운 AI 추천 유형이 생기면: `AI 개인화`와 `결과` 섹션
- 토큰이 바뀌면: `비주얼 시스템`
