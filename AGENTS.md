# AGENTS.md

## 목적
이 저장소에서 UI/UX를 수정하는 모든 작업은 `DESIGN.md`를 단일 소스 오브 트루스로 삼는다.  
목표는 **중구난방인 화면을 줄이고, 온보딩→인증→레벨테스트→AI 개인화→메인 사용 흐름을 일관되게 유지하는 것**이다.

> 디자인 기준은 Google Stitch의 `DESIGN.md` 문법과 `awesome-design-md`식 “한 제품 = 한 일관된 미감” 철학을 따른다.

## 제품 관점 요약
- Batter’s Eye는 **모바일 야구 시야 훈련 앱**이다.
- 핵심 경험은 **짧은 세션, 빠른 피드백, 다음 행동 제안**이다.
- 감성은 **UIBowl식 정돈된 카드 UI + The Ump Show식 게임 긴장감**의 조합이다.
- 전체 앱은 차분하고 프리미엄해야 하며, 게임 느낌은 훈련 순간에만 강해진다.

## 반드시 지킬 구조

### 1. 앱 레벨 구조
- 온보딩은 **독립 풀스크린 플로우**로 유지한다.
- 메인 앱은 **홈 / 훈련 / 결과 / 설정** 4개 축으로 정리한다.
- 새 화면을 추가할 때는 먼저 이 4개 축 중 어디에 속하는지 명시한다.

### 2. 권장 사용자 플로우
- 첫 방문: `인트로(3슬라이드) → 인증 → 프로필 → 레벨 테스트 → AI 개인화 → 홈`
- 재방문: `홈 → 훈련 → 결과 → 홈 또는 결과 탭`
- 설정은 보조 축이며 훈련 플로우를 방해하지 않게 유지한다.

### 3. 화면 목적 규칙
한 화면은 아래 중 하나의 목적만 강하게 가져야 한다.
- 시작시키기
- 선택시키기
- 측정하기
- 해석해주기
- 설정 바꾸기

두 개 이상이 강하게 섞이면 화면을 분리한다.

## UI 시스템 작업 규칙

### 1. 토큰 우선
- 색, radius, spacing, 타이포는 `DESIGN.md`와 `app_theme.dart` 기준으로 맞춘다.
- 카피는 `app_copy.dart` 같은 단일 레이어에서 관리하고, 화면 안에 문자열을 흩뿌리지 않는다.
- 화면 안에서 임의 hex, 임의 radius, 임의 패딩을 늘리지 않는다.
- 새 스타일 값이 필요하면 먼저 토큰 승격 가능성을 검토한다.

### 2. 공통 컴포넌트 우선
새 UI를 만들 때는 아래 재사용 단위를 먼저 고려한다.
- Page Header
- Hero Card
- Mode Card
- Metric Card
- Coach Card
- Progress Strip
- Result Insight Block

유사한 UI를 새로 복제하지 말고 공통화 방향을 우선 검토한다.

### 3. CTA 규칙
- 화면당 Primary CTA는 1개만 둔다.
- Secondary CTA는 보조 행동에만 사용한다.
- 로그아웃, 뒤로가기, 부가 설정은 Primary처럼 보이면 안 된다.

### 4. 카피 규칙
- 한국어 우선
- 짧고 직접적인 코치 톤
- 숫자는 항상 해석 또는 권장 행동과 함께
- 생산성 앱/자기계발 앱 같은 일반 문구 금지

## 화면별 체크리스트

### 홈
반드시 있어야 하는 것:
- 오늘의 추천 또는 바로 시작 Hero
- 최근 지표 요약
- AI 추천 이유
- 최근 결과 또는 다음 단계

피해야 하는 것:
- 설정 기능 과다 노출
- 모든 기능을 한 화면에 동일 비중으로 배치

### 훈련
반드시 있어야 하는 것:
- 현재 모드/라운드/진행률
- 명확한 선택지
- 즉시 피드백
- 세션 종료 후 결과 진입

피해야 하는 것:
- 긴 설명문
- 입력보다 읽기가 많은 UI
- 과도한 장식 모션

### 결과
반드시 있어야 하는 것:
- 핵심 메트릭
- 약점 해석
- 다음 추천 훈련
- 누적 기록으로 이어지는 연결점

피해야 하는 것:
- 수치만 있고 해석이 없는 화면
- 결과 확인 후 막히는 플로우

### 설정
반드시 있어야 하는 것:
- 계정/프로필
- 테마/언어
- 알림/데이터 관련 옵션

피해야 하는 것:
- 홈에서 다뤄야 할 핵심 기능을 설정으로 숨기는 것

### 온보딩
반드시 있어야 하는 것:
- 현재 단계 표시
- 입력 최소화
- 레벨 테스트 완료 후 AI 개인화 진입
- 첫 훈련 시작 CTA

피해야 하는 것:
- 단계 누락
- 저장 완료 전 홈 진입
- 개인화 근거 없는 추천

## AI 개인화 규칙
- AI 추천은 반드시 **근거 데이터**를 함께 보여준다.
  - 예: 레벨 테스트 결과, 최근 정확도, 반응속도, 약점 존
- AI 실패 시 fallback 문구와 기본 훈련 플랜을 제공한다.
- AI는 결과를 “예쁘게 포장”하는 용도가 아니라 **다음 행동을 정하는 용도**여야 한다.
- 기본 모델 타깃은 **Gemini 2.5 Flash Lite**다.
- API 키가 없을 때는 deterministic fallback planner로 동일한 카드 구조를 유지한다.
- AI 카드에는 반드시 아래 4개 중 최소 3개가 있어야 한다.
  - 현재 강점
  - 현재 위험 / 약점
  - 왜 이 모드를 추천하는지
  - 첫 7일 액션 플랜

## 3D / 현실감 규칙
- 공 궤적과 릴리스 표현은 앱의 핵심 가치이므로 일반 카드보다 우선순위가 높다.
- 단, 전체 앱을 Unity로 바꾸지 않는다. **필요하면 pitch renderer 구간만 모듈형으로 분리**한다.
- 1차 목표는 Flutter에서 faux-3D/깊이감/속도감 개선, 2차 목표는 Unity/SceneKit류 전용 렌더러 검토다.
- 현실감 개선은 항상 입력 속도와 정확도를 해치지 않아야 한다.

## Hyper-Waterfall Lite 운영 규칙
이 저장소는 `mydocs/manual/hyper_waterfall.md`를 따른다.

### 언제 적용하나
- 새 사용자 플로우 추가
- 아키텍처/상태 전이 변경
- 외부 API/AI 모델/패키지 선택
- 디자인 시스템 전면 수정
- 3D/Unity 같은 렌더링 경계 변경

### 기본 절차
1. `mydocs/orders/YYYYMMDD.md`에 작업을 등록한다.
2. `mydocs/plans/task-<slug>.md`로 목표/비목표/검증 기준을 고정한다.
3. 범위 확정 또는 승인 게이트를 거친다.
4. 구현 후 `mydocs/reports/`에 stage 또는 final 보고를 남긴다.

### 승인 게이트
다음은 바로 구현하지 말고 먼저 계획을 보여준다.
- 파일 3개 이상 수정
- 사용자 플로우 변경
- 패키지/빌드 체인/스크립트 변경
- 외부 모델/API 연동 변경
- 대규모 UI 방향 전환

## npm / shell 보안 규칙
> 중요: 이 저장소에서는 `npm install`을 쓰지 않는다.

- 문서, 스크립트, 예시 명령어, AI 응답 모두 `npm ci`를 사용한다.
- `.sh` 스크립트에서 Node 의존성을 설치할 일이 생기면 `npm ci`로 작성한다.
- lockfile 없이 `npm ci`가 불가능한 경우, 임의로 `npm install`로 바꾸지 말고 먼저 사용자 승인 또는 별도 계획 문서를 만든다.
## 문서 업데이트 트리거
아래 변경이 있으면 `DESIGN.md`와 이 문서를 같이 점검한다.
- 새 탭/새 최상위 화면 추가
- 온보딩 단계 변경
- 디자인 토큰 변경
- 공통 컴포넌트 추가 또는 제거
- AI 추천 로직/표현 방식 변경

## OMX / Codex / MCP 운영 메모
- OMX 0.17+를 기본 전제로 본다.
- 디자인 작업은 `DESIGN.md` + 이에 대응하는 `$design` 워크플로를 canonical로 취급한다.
- Hermes MCP bridge는 **bounded / opt-in coordination layer**로 취급하며, 세션 상태 확인·후속 작업 디스패치·artifact 읽기 같은 안전한 조정 작업에만 사용한다.
- tmux continuation이나 stale pane 재사용보다 현재 저장소 문서(`mydocs/`, `DESIGN.md`, `AGENTS.md`)를 우선 신뢰한다.
- UltraQA 성격의 검증이 필요할 때는 interrupt/cancel/resume, stale state, cleanup evidence까지 체크한다.
- plugin-mode discovery로 OMX skills가 더 잘 노출되더라도, 이 저장소의 단일 설계 기준은 여전히 `DESIGN.md`다.

## 작업 전 자가 점검
UI 작업 전에 아래를 확인한다.
1. 이 화면은 4개 축(홈/훈련/결과/설정) 중 어디에 속하는가?
2. 이 화면의 Primary CTA는 무엇인가?
3. 같은 역할의 UI가 이미 다른 화면에 존재하는가?
4. 지금 추가하는 정보가 정말 이 화면에 필요한가?
5. 결과적으로 사용자가 더 빨리 훈련을 시작하거나 이해할 수 있는가?

## 작업 후 자가 점검
- 헤더 스타일이 다른 화면과 일관적인가?
- 패딩, 라운드, 컬러 사용이 토큰과 맞는가?
- 숫자에 해석이 붙어 있는가?
- 빈 상태/오류 상태/로딩 상태가 정의되어 있는가?
- 게임 느낌이 훈련 구간 밖으로 과하게 새지 않았는가?

## 금지사항

- 디자인 문서 미반영 상태로 UI 구조 변경
